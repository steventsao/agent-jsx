## cloudflare/agents (v0.8.5)

1. **`this.name` throws under direct DO access.** partyserver's `name` getter throws unless the name was set via `getAgentByName`/`setName`/routing (`partyserver/src/index.ts:601`, workerd issue #2240) — and the agents pkg reads `this.name` internally in `_emit` on **every `setState`** (`agents/src/index.ts:853`, `:1402`). Consequences encoded:
   - Generated callback refs route by **`ctx.id` string** (reconstructable via `idFromString`), never `this.name`.
   - Tests reach DOs via `getAgentByName` — the production path — not raw `env.NS.get(idFromName(...))`.
2. **`onStateUpdate` is deprecated; the hook is `onStateChanged`** (`agents/src/index.ts:685` — "new → call onStateChanged, old → call onStateUpdate (deprecated)", warn-once at `:500`). The template uses `onStateChanged` with a changed-guard so `reconcile → setState` can't loop. The spawn test asserts the full production loop (`setState → onStateChanged → reconcile → child spawn`) with no manual `reconcile()`.
3. **Raw `setState` is full-replacement** — it wipes the runtime's `__children` bookkeeping. Production writes flow through the generated merging `boundStore` (partial updates), so this only bites external writers. The agents pkg protects its own internals with hidden `_cf_`-prefixed keys (`agents/src/index.ts:673`); moving `__children` into `this.sql` or a hidden key is the proper fix (future work).
4. **A throwaway `createStore(state)` drops handler writes.** Handler closures call `store.set(...)`; the store must be bridged to the DO (`get: () => this.state`, `set: → this.setState`) or sensor/callback state changes silently vanish. (codex)
5. **Schedule convergence replaces idempotency bookkeeping.** The pkg patches duplication per-call (`{ idempotent: true }`, cron-idempotent-by-default, `onStart()` warning about "accumulating duplicate rows across Durable Object restarts", `agents/src/index.ts:2308-2363`). The generated reconcile instead diffs desired schedules against `getSchedules()` by a payload key — the idempotent-wake test passes with zero per-call options.
6. **Copied components need import rewriting.** Authored components import `../src/state.ts` (the React hook build); compiled packages rewrite onto the emitted react-free runtime (`state.ts → store.ts`, whose `useAgentState` is a plain read) via `copyAgentComponent` (`src/compile/runtime-files.ts:49`). React never enters the worker bundle — `import type { ReactNode }` erases.
7. Minor: `cron-schedule` ships a stale sourcemap (harmless warning in vitest output).

## flue v0.5 — the reactive workflow executor

The v0.5 completer is a generated `defineWorkflow` whose `run()` drives a
react-free `runReactiveWorkflow` (`src/workflow-executor.ts`, shipped as
`runtime/workflow-executor.ts`). Wiring it against the REAL `defineWorkflow`
shook out these divergences from the first-draft assumptions.

12. **`defineWorkflow` folds `run()` into an `action`; there is no top-level `.run`.** The RED workflow test assumed `workflow.run(ctx)`. Reality: `defineWorkflow({ agent, input, run })` calls `defineAction({ name:'workflow', input, output, run })` and returns a **frozen** `{ __flueWorkflowDefinition, agent, action }` (`flue/packages/runtime/src/workflow-definition.ts:52-92`, `:94-105`). The runnable is `workflow.action.run` (`action.ts:29-39`, run at `:38`). Because the object is `Object.freeze`d and identity-tracked in a `WeakSet` (`isWorkflowDefinition`, `workflow-definition.ts:98-109`), the module's default export MUST be the `defineWorkflow` result verbatim — you can't wrap it to re-expose `.run`. So the test's harness plumbing was corrected `workflow.run(...)` → `workflow.action.run(...)` (sanctioned API-shape fix, behaviors unchanged) and strengthened to assert `__flueWorkflowDefinition === true` and `typeof action.run === 'function'`.
13. **`run(context)` gets an `ActionContext = { harness, log, input }`, not `{ harness, input }`, and there is no top-level `session`.** `ActionContext` is `{ harness: FlueHarness, log: FlueLogger } & { input }` (`action.ts:18-23`); the runtime invokes `action.run({ harness, log })` (`action.ts:126-147`, run at `:132`). The session is obtained via `context.harness.session()` → `Promise<FlueSession>` (`types.ts:462-469`). The emitter's `run()` does `const session = await context.harness.session()` before delegating — the earlier `harness.session()` hint was directionally right but `session` is not handed in.
14. **Workflow `input` must be a TOP-LEVEL object schema, and flue does not re-export valibot.** `defineAction` throws unless `isTopLevelObjectSchema(input)` (`action.ts:83-85`), which accepts `object`/`strict_object`/`loose_object`/`object_with_rest` (`schema.ts:42-45`) — so `v.object(...)` qualifies, `v.optional(...)`/`v.any()` at top level would not. `@flue/runtime`'s index exports `defineWorkflow` but NOT `v`/valibot (`index.ts:152`), so the emitted module imports `* as v from "valibot"` directly (valibot resolves transitively from `compat/flue`). The reactive entry state is arbitrary, so it is nested as `input: v.object({ state: v.any() })` — the top level stays an object while `state` carries any shape.
15. **Delegation is `session.task(text, { agent }) → PromptResponse { text }`.** `FlueSession.task(text, options?)` takes `TaskOptions.agent?: string` selecting a named subagent profile and resolves to `{ text, usage, model }` (`types.ts:552-556` task, `:674-683` TaskOptions, `:620-627` PromptResponse). The executor's `delegate` maps `SpawnDescriptor.agent` (the `<subagent kind>`) onto `{ agent }` and folds `response.text` back through the record's `onResult`. The compat test's fake harness fakes ONLY this transport (never `@flue/runtime`) and mirrors the shape exactly, with the citation inline.
16. **`@flue/runtime/test-utils` cannot run a workflow headless.** Its `./test-utils*` subpath exports are storage-adapter CONTRACT-TEST factories — `define-store-contract-tests`, `define-conversation-stream-store-contract-tests`, `define-attachment-store-contract-tests` (`packages/runtime/package.json` exports) — not a scripted-model harness. There is no headless real-model runner, so the sanctioned transport-fake path (#15) is the honest one; `@flue/runtime` itself is never mocked (defineWorkflow/defineAgent/defineAgentProfile all run for real).
17. **The executor's result round-trips as the workflow output.** `runReactiveWorkflow` returns `{ state, rounds, delegated, prompt }`, returned verbatim from `run()`. Under the real runtime `runActionWithParsedInput` `cloneJsonSerializable`s the output (`action.ts:142-145`); every field is JSON-serializable, so no wrapper is needed — the executor's return value IS a valid workflow return (COMPAT.md's "adjust the emitted wrapper not the executor" escape hatch went unused). The **v0.5 parity theorem holds**: from the same incident state the executor delegates the same stableIds (`["investigate:https://b.example"]`) and produces byte-identical final state to the live SimHost path (`tests/workflow-executor.test.tsx`), because the descriptor `{ stableId, agent, input }` and the `onResult` fold mirror `spawnPlan` + SimHost's canned result exactly.

## v0.6 — method props: request/response RPC with return values

A function prop is a capability *with a return value*: the child awaits
`props.lookupRunbook(site)` like a local call while the parent's freshest
closure computes the answer from parent state. Greening the RED contract
(`tests/emit-method-props.test.tsx`, `compat/cloudflare/test/method-props.spec.ts`)
shook out these facts about DO RPC.

21. **A method prop round-trips over native Durable Object RPC, so args AND return must be structured-cloneable.** The child's proxy calls `parent.onAgentEvent(...)` on a `DurableObjectStub` (reconstructed via `idFromString`) and returns its result; the parent's dispatcher returns the invoked closure's awaited value. Our generated classes extend `Agent → Server → DurableObject` (`~/dev/cloudflare-agents-playground/packages/agents/src/index.ts:665` `class Agent<…> extends Server`; `partyserver/dist/index.js:367` `var Server = class extends DurableObject`), so `onAgentEvent` is a plain DO-RPC method — workerd JSRPC **structured-clones both the call args and the return value** across the stub. Hence a method prop's arguments and its return value must be plain data (strings/numbers/objects/arrays), never functions, symbols, or non-`RpcTarget` class instances (those throw `DataCloneError`). The demo's `lookupRunbook(site: string): string` satisfies this: string in, string out. This is the concrete meaning of the v0.6 scope note "args/returns must be structured-cloneable" — a capability needing to hand back a *live* handle would have to return an `RpcTarget`/stub, not a value.

22. **Two template fixes turn fire-and-forget into request/response; DO input-gate ordering keeps the void path correct.** (a) The child proxy now does `return await parent.onAgentEvent(...)` and the dispatcher's callback branch now `return await`s the routed `subagent:<child>` handler instead of invoking it and `return`ing void — so the parent's freshest closure result crosses back. A missing/void handler resolves to `undefined` via the optional chain, which is exactly how `onResult` stays fire-and-forget while `lookupRunbook` yields a value. (b) The dispatcher now **`await`s `onFire`/`onStatus`**: the child's `sla-deadline` `onFire` is now `async` and `await`s `lookupRunbook` (a cross-DO RPC), so if the dispatcher didn't await it, the firing DO's `onAgentEvent` would resolve while `onFire` was still suspended on that RPC — a hanging promise whose continuation (`onResult`) could be dropped as the I/O context unwinds. The un-awaited `onResult` *inside* `onFire` still lands before the test's subsequent parent read because a Durable Object serializes stub calls under its input gate: the `onResult` RPC is enqueued at the parent before the read RPC, so the fold is already visible (this is also why the pre-v0.6 synchronous `onResult` path worked — #19). Proven in real workerd: `method-props.spec.ts` sees the finding carrying both the runbook text and the parent-state-derived `statuses tracked: 1`, never `undefined`. `uptime.spec.ts` stays 5/5 — its callback test now flows through the awaited runbook too.

23. **Return-value semantics are CF-DO-specific; the flue/sim boundary stays one-shot.** `emit-flue.ts` is untouched. flue delegation is a one-shot `session.task(text, { agent }) → { text }` (#15): parent props become the task input and the child's single result returns as the task text — there is no channel for the child to call synchronously *up* into a running parent and await a computed value mid-task, so the generated flue child keeps framing the callback as "the task RETURN value." `SimHost` likewise folds opaque child records through a canned `onResult` (v0.5 parity theorem, #17), so a method prop is inert there. Capability-with-return-value therefore lives on the CF target (and any future live child mount speaking the same DO RPC), exactly as the v0.6 scope note states.

24. **Live redeploy verified the round-trip on production Cloudflare (2026-07-07).** Redeployed `agent-fiber-demo` from the v0.6 emitter (version `99de8c46`, 1255 KiB / 229 KiB gzip). The prior deploy had left a stale runbook-less finding (`[…] no root cause within SLA — escalating` — the old fire-and-forget `onResult` path), which is expected: the child's persisted `__callbacks` predated `lookupRunbook`. `POST /recover` then `POST /incident` on `https://agent-fiber-down.invalid` respawned a fresh child adopted with the new `{ onResult, lookupRunbook }` callbacks; `/state` then showed the fold `[https://agent-fiber-down.invalid] no root cause within SLA — escalating (runbook: restart edge pods for https://agent-fiber-down.invalid; statuses tracked: 1)`. Both the runbook text (parent's live closure) and the parent-state-derived `statuses tracked: 1` are present, never `undefined` — the child awaited `lookupRunbook` over DO RPC and the parent's freshest closure computed the answer from parent state, in production. (Redeploying `agent-fiber-demo` was the only pre-authorized deploy; nothing else was deployed.)

## rename + the self-identity regression (agent-jsx)

33. **Never derive durable identity from partyserver's `.name`.** It is request-scoped: set by `getAgentByName` routing, throwing (or varying) on unnamed wakes — alarms, `onStart` after eviction. Commit a8b2782 wove raw `this.name` into child DO names, the `__children` roster, and `CallbackRef.parentName`; any cross-context variance renamed the entire child fleet each reconcile → simultaneous `setProps` + `shutdown` cycling, live-only (under vitest-pool-workers every access flows through `getAgentByName`, so the name never varies and the suite stays green). Fix: generated `selfName()` persists identity into state (`__self`) on first named access; unnamed wakes read the copy or DEFER child convergence rather than minting unstable names. Diagnosed by codex during the rename sweep via old-vs-new worker bisection (old emitter build converged; rebuilt one looped — the failure tracked the emitter version, not the name).

34. **`onStart` must not reconcile inline — blockConcurrencyWhile deadlock.** The Agents SDK wraps `onStart` in `blockConcurrencyWhile` (input gate closed). If a DO wakes holding pending pipeline state, an inline reconcile spawns children whose immediate RPCs back into the waking DO (`getPdf`, `onSegment`) cannot pass the gate: 30s cancel → "Durable Object was reset" → next request wakes it again → permanent 1101 loop, and the instance looks poisoned across REDEPLOYS (state persists; every build wakes into the same trap — which is why the "old" worker seemed fine: its state was completed, nothing to spawn). Normal RPC contexts open the gate at await points, which is why `/run` always converged and why the entire vitest matrix stays green (miniflare doesn't reproduce the gating). Fix: `onStart` schedules an immediate alarm tick (`schedule(0, "onAgentEvent", {key:"__wake-reconcile"})`) — alarms run gate-open. Diagnosis chain: codex's old-vs-new worker bisection (#33's naming hazard was real but not sufficient) → `wrangler tail` on a GET revealing the blockConcurrencyWhile cancellation.

35. **`onStateChanged` must REQUEST, never await, the in-flight reconcile.** a8b2782's single-flight wrapper let a callback-triggered `onStateChanged` await the very pass that is blocked awaiting the callback's own chain (parent reconcile #1 → `child.setProps` → child work → parent `setState` → `onStateChanged` → await reconcile #1's promise): a distributed await-cycle across DOs. Mark-and-return (`#requestReconcile`) + the drain loop preserve convergence with no cycle.
36. **a8b2782's child template made children externally driven — composed pipelines need self-driving children.** `onStart`/`onStateChanged` were no-ops and the comment deferred all work to `POST /api/drive`; without that driver, live children adopted props and sat inert (the final layer of the stall — vitest flows masked it). `setProps` now self-schedules an immediate gate-open alarm tick after adoption: honors the no-inline-work-during-adoption constraint AND needs no external driver; an explicit /api/drive remains idempotent alongside it.

**Live proof after #33–#36 (2026-07-08):** `agent-jsx-pdf-compiled` POST /run → 4/4 segments diff-equal to the golden oracle on the previously poisoned "main" instance — wake self-heal + full convergence on the HEAD emitter.

## continuation nesting on real workerd (agent-jsx)

The continuation feature (function-as-children: a child `emit`s → parent `__outputs`
→ the parent's render-prop continuation fans out grandchildren) was proven only at
the unit/parity/emitted-string level. Greening a real end-to-end compat proof on
workerd — a minimal PURE-COMPUTE pair (`examples/continuation-min/`: cont-root nests
cont-emitter statically, fans out cont-folder grandchildren) exported into
`compat/cloudflare` and driven through THREE DOs by `continuation.spec.ts` — shook out
two facts.

37. **A continuation `emit` — and any fold-back callback fired from a `<task>` onDone — is a cross-DO RPC and MUST be `await`ed inside onDone.** The generated child's `emit` compiles to `proxied.__emit`, an `async` call that RPCs `parent.onAgentEvent({callback:{event:"__emit"}})`; the fold-back `onFolded`/`onSegment`/`onResult` are the same shape. The task loop `await`s `onDone`, but an onDone that fires the callback fire-and-forget (`emit?.(x)` with no await) returns before the RPC settles. When the compat proof drives the child synchronously — `runInDurableObject(child, c => c.reconcile())` and reads immediately — that pending RPC is still in flight when the harness closes the I/O context: workerd throws `EnvironmentTeardownError: Closing rpc while "resolve" was pending` (at partyserver `getServerByName`, from the emit's `getAgentByName`), and the parent's continuation never expands (grandchildren absent). Fix: the emitting onDone is `async` and `await`s the emit / fold-back — the exact await-the-cross-DO-call rule the dispatcher already honors for `onFire`/`onStatus` (#22). Awaiting is deadlock-safe because a DO's input gate OPENS at its cross-DO await points (#34), so the parent's re-convergence `setProps` back into the still-awaiting child proceeds rather than blocking. The `AgentImpl` `emit` type is widened to `() => void | Promise<void>` to reflect this (inert/synchronous store-write in sim, awaitable RPC on CF). NOTE the alarm-driven pipeline (#32, pdf-compiled `onSegment`) tolerates fire-and-forget because the wake-tick context outlives the RPC and DO input-gate ordering lands it before the next read; awaiting removes that dependency and is the robust pattern for any synchronously-driven proof.

38. **The emit lands `__outputs` synchronously, but the continuation fan-out is a SEPARATE, non-awaited reconcile — drive it explicitly to converge.** The `__emit` handler's `#applyOutput` writes `this.state.__outputs` and REQUESTS (never awaits, #35) the reconcile that expands the continuation; `setState` also fires `onStateChanged`, whose reconcile grabs the single-flight lock first, so the `if (needsReconcile) await #requestReconcile()` in the callback branch marks-and-returns without awaiting that in-flight pass. So when the emit RPC resolves, `__outputs` is durable but the grandchildren may not be spawned yet (the fan-out floats to completion). Exactly like uptime.spec drives an explicit `reconcile()` because vitest-pool-workers does not auto-fire alarms, `continuation.spec.ts` drives ONE drained `parent.reconcile()` after the emit to converge the fan-out deterministically before asserting (deployed, the emitter's post-emit wake tick drives it). Both the child's OWN state (`folder.state.folded`) and the fold-back into parent state (`parent.state.folded`) are then visible — the whole round-trip across three real DOs.

**Live/compat proof (2026-07-08):** `compat/cloudflare` `bun run test` 7/7 in real workerd (vitest-pool-workers) — the 6 base/method-props specs plus `continuation.spec.ts`: applyState(seed) spawns the emitter → the emitter's `<task>` `emit`s in its own DO → reserved `__emit` RPC lands `["a","b"]` in the parent's durable `__outputs` → the parent re-render fans out `fold:a`/`fold:b` grandchildren → each folds its own uppercased result (visible in its own DO) back into `parent.state.folded === {a:"A", b:"B"}`.
